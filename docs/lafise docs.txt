API Overview
Welcome to LAFISE Poket API documentation. LAFISE Poket API is a cutting-edge financial solution that leverages on our vast banking experience over the past 30 years. With this API, LAFISE brings its expertise to the digital age, providing customers with easy-to-use, secure and reliable access to a range of financial services.

Note:
This documentation is for developers and technical managers who want to integrate with our services to offer a financial solution. For program details, rules, and further implementation guidance, please contact your LAFISE representative.

📝 Poket Overview
With our API, businesses can leverage our extensive experience in the financial industry to offer their customers a comprehensive suite of banking services, while reducing development time and costs. Our API supports a wide range of use cases, from simple balance inquiries to more complex financial transactions, providing businesses with the flexibility and scalability needed to stay competitive in today's digital landscape.

📢 Who can use it
Our Poket API enables third-party developers, startups, and fintechs to seamlessly integrate our financial products and services into their digital platforms.

📖 How it works
We offer various endpoints such as Account Information, Payment Initiation, Fund Transfer, Payment Status, Transaction History, and Account Holder, which allow users to manage their financial activities with ease. LAFISE's API utilizes advanced security measures, including OAuth 2.0 Authentication, SSL Encryption, Rate Limiting, and User Permissions, to ensure the security and stability of the platform. With LAFISE Poket API, our customers can access financial services seamlessly and efficiently, thus helping them to manage their finances more effectively.

LAFISE Poket is organized around REST and uses built-in HTTP features, standard response codes, authentication and verbs, allowing for quick implementation with standard HTTP packages used in all languages. The API accepts requests with a JSON body and returns JSON responses.

🔑 API Credentials
To start using our API, you must first obtain credentials for your team, interested parties need to become an Poket Allied by contacting our Poket Corporate Banking Office. Our team will provide all the necessary information and support to help customers become an Poket Allies and gain access to the API. At LAFISE, we are committed to providing our customers with exceptional service and support, and our Corporate Banking team is always available to assist with any queries related to our Poket API. So, if you're interested in accessing our API and becoming an Poket Allied, please reach out to our Corporate Banking team today.

📋 Next steps
Once a relationship with LAFISE has been established, customers will receive three key pieces of information: a client_id, a secret, and an API key. These credentials are essential for accessing and using our Poket API, and will be provided to customers by our Corporate team.


Once you have your credentials, to authenticate a request, you must first generate an Authorization Token using the "Authentication" API endpoint.

✅ IP Address Whitelist
As an additional security layer, LAFISE Poket API requires that your application's IP address be whitelisted before you can use the API. This means that we will only allow requests from the specific IP addresses that you have provided to us. By doing this, we help ensure that only authorized applications are accessing the API, thereby enhancing the security of our platform and protecting user data.

To whitelist your application's IP address, please contact our Corporate Banking team and provide the necessary details. Our team will guide you through the process and ensure that your application is granted access to the API as soon as possible. At LAFISE, we take security very seriously, and this additional layer of protection is just one of the ways we ensure that our customers' data is safe and secure when using our Poket API.

🌎 Regional Scope API
Our Poket API is regional, covering Central America. To make requests to the API, you must specify the country code where your customer is located. The scope of the API currently covers five countries: Nicaragua (BLNI), Honduras (BLHN), Costa Rica (BLCR), Panamá (BLPA), and Dominican Republic (BLRD). This is an important feature as it allows you, as a developer, to target all the countries where LAFISE is present with a single integration to our API, streamlining the development process and reducing time-to-market for your products and services.

📝 Glossary
Term	Description
Access Token	A token which is retrieved by the Client after a successful access authorization flow. The access token is used by the API to authenticate Clients. The access token does not contain any information that the Client (or client's application) can use.
API	Application Programming Interface. A set of definitions, protocols, and tools that can be used to create applications, interact with other applications, and exchange data.
API Call	API call is a request towards an API which produces a response. The API is by design stateless, and therefore it does not "remember" anything about previous requests, i.e., there is no session. Therefore every request made towards the API must contain certain headers so that the API can authenticate and authorize the client.
ASPSP	Account Servicing Payment Services Provider. An entity authorized to operate customer accounts, with a line of credit and payment facilities online.
Authorization	Authorization is the process through which the Client obtains permission to access the Resource Owner's data and services at the bank.
Authorization Code	An authorization code is a code provided to the Client during the access authorization flow; this code is short-lived and exchanged for an Access Token.
Authorizer	Authorizer - Authorizers are strictly required for both account access and payment initiation requests. The minimum number of Authorizers is 1 although the customer may demand that two or more Authorizers act together, on the Resource Owner's behalf, to confirm any request.
Authorizer Authentication	Authorizer authentication: the process which provides the correct identity of an Authorizer acting to confirm the access request or payment confirmation on behalf of the Resource Owner.
Client	The client refers to the consumer of the API. This is commonly an application provided by the corporate customer or a regulated third-party provider.
Consent	Consent is agreed between the Resource Owner (customer) and the Client (TPP). It includes what data may be shared, what services may be performed on the Resource Owner's behalf, the duration of the agreement and to what purpose the data/services are provided. Consent agreed between Resource Owner and Client, informs the access authorization flow.
Sandbox	Sandbox, in the context of the Corporate APIs, is a test environment facility where the data returned by the API consists of example data. Its purpose is to mimic the real / production version of the API. The sandbox will always have available the latest published version of the API, this means that all new versions appear in sandbox before they are introduced into the production.
Third-Party Provider (TPP)	A TPP is a regulated provider of one or more applications which Resource Owners (customers) can use to access their accounts/services via the API. A TPP is one type of API client/consumer. See also Corporate Direct Access.


Using OAuth2

OAuth 2 is an authorization mechanism that gives you permission to a) Call LAFISE APIs and b) Access the end user’s data (given the user’s consent). This permission is encoded in an access token that you obtain through the OAuth operation "Authentication" that is described in this document. When you call a LAFISE API, you pass the access token to the Authorization header and the x-api-key in the request message, like this:

El consumidor
Un consumidor es toda aquella aplicación que consuma las APIS de creación de Link de Pago, siguiendo el estándar OpenBanking establecido de integración de terceros. El consumidor es el que en última instancia se encarga de definir las reglas de negocio para crear o actualizar un enlace, llevar control interno de los enlaces creados y relacionarlo con sus propias entidades, y aplicar lógica de negocio ante los eventos de pago, delegando al sistema de enlaces de pago la lógica de ejecución de un pago y su notificación al consumidor.

Creación de un enlace pago
El enlace de pago debe ser creado mediante el endpoint correspondiente, brindando la información necesaria según los métodos de pago a habilitar (por ejemplo, pago con tarjeta necesita del terminal procesador o por transferencia necesita el número de cuenta a acreditar). El consumidor debe manejar su lógica de negocio y validaciones previa a la creación del enlace según corresponda, almacenar la información y relacionar a sus entidades, y presentar o redirigir al pagador al sitio de pago.


⚠️
Sobre la seguridad
El consumo de las APIS de Link de Pago debe hacerse exclusivamente desde un backend y no directamente desde el cliente, según indica el tipo de seguridad OAuth2 bajo las que se integra. Leer más


Integración de eventos (webhook)
Una vez generado un enlace de pago, este está listo para ser pagado por cualquiera de los métodos configurados. Esto genera dos eventos: inicio de intento y final de intento

Eventos de pago
Un intento de pago se inicia desde el momento en el que el pagador dentro de nuestro sitio confirma el método de pago, y el sitio le redirige donde corresponde para completar el pago (por ejemplo, si escoge pagar con tarjeta se le redirige a un formulario donde completará la información de la tarjeta); y se termina en el momento que el método de pago termina de procesar, pudiendo ser exitoso o fallido.


ℹ️
Sobre la seguridad de los webhooks
Los webhooks pueden ser accesibles via internet o VPN (requiere configuración adicional de conectividad por parte de LAFISE). Se soporta autenticación Basic y OAuth2 con client credentials, y API KEY.

Inicio del pago
Al momento de iniciar un intento, el consumidor puede configurar un endpoint que funcione como webhook para realizar validaciones y reglas de negocio propias.

Si se rechaza el intento de pago, el webhook debe responder con código HTTP 400 (BAD REQUEST), de lo contrario si se acepta, debe responder HTTP 200 OK.

Final del pago
Al finalizar un intento de pago, el consumidor puede configurar un endpoint que funcione como webhook para recibir la notificación de intento finalizado. El intento puede ser exitoso o fallido, en ambos casos será notificado de igual manera, agregando información correspondiente al estado (por ejemplo, agregando el motivo de error en caso fallido o agregando información del pagador en caso exitoso).

Un pago es exitoso cuando el método de pago procesa correctamente.

Un intento es fallido cuando el método de pago rechaza el pago por cualquier motivo (por ejemplo, tarjeta sin fondos, bloqueada, expirada, transacción insegura) o cuando el pagador sobrepasa una determinada cantidad de tiempo sin interacción luego de iniciar el proceso (en este caso se declara el intento fallido por timeout-no payer response). Un pago fallido puede ser reintentado hasta completar la cantidad máxima de intentos configurados al momento de crear el enlace.

⚠️
Sobre el proceso propio del webhook de final de pago
Recomendamos este proceso a ejecutar al finalizar un intento corto y conciso. Si el proceso es complejo o involucra a más actores, recomendamos ejecutar una acción asíncrona al invocar el webhook.

El llamado al webhook del consumidor no impacta la experiencia del pagador en nuestro sitio y la respuesta de este endpoint no es utilizada por nuestro proceso.

Este endpoint permite actualizar un enlace de pago asociado a un consumidor autenticado. Además de modificar los datos principales del enlace de pago, tambien valida y actualiza su configuración asociada ("LinkConfiguration").

La operación consiste en:

Verificación del consumidor actual mediante "AppClientId". Si no se encuentra, lanza una excepción.
Búsqueda del PayLink por su ID. Si no existe, lanza una excepción.
Validación de que el PayLink pertenece al consumidor actual; si no, lanza una excepción.
Si se proporcionan configuraciones en la solicitud, se valida que los métodos incluidos estén permitidos por el consumidor.
Se mapean los datos del modelo de entrada al PayLink existente, actualizando también la lista de métodos asociados.
Se actualiza el PayLink y se persiste en base de datos.
Se busca y actualiza la entidad LinkConfiguration correspondiente al PayLink:
Si no se encuentra, lanza una excepción.
Se actualiza el método y la configuración serializada en JSON.
Finalmente, se retorna un DTO ("UpdatePayLinkDto") con los datos actualizados del PayLink.

Este endpoint permite realizar la eliminación lógica de un enlace de pago ("PayLink") identificado por su "PayLinkId".

La operación consiste en:

Verificar que el enlace de pago exista.
Cambiar su estado ("Status") a Cancelled, reflejando que el enlace ha sido anulado.
Busca su configuración ("LinkConfiguration") y si existe la elimina.
⚠️
Advertencia
Esta acción no elimina físicamente el registro del "PayLink" en base de datos, pero lo marca como desactivado para evitar su uso posterior.
